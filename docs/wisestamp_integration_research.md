# WiseStamp Integration Research Notes

## Official sources reviewed

1. [WiseStamp Integration Features](https://www.wisestamp.com/features/integrate/) describes WiseStamp as integrating with major email platforms, including Google accounts, Outlook services, Mac Mail, and iPhone/iPad. It mentions API connections in the context of Gmail and Google Workspace compatibility, but does not publish a public REST endpoint or authentication scheme for retrieving a rendered user signature.
2. [WiseStamp Signature Deployment Methods](https://support.wisestamp.com/en/articles/12561463-wisestamp-signature-deployment-methods) says WiseStamp supports client-side, server-side, and hybrid deployment. Its server-side mode applies a signature after an email is sent, routes email through WiseStamp via the organization's Google or Microsoft mail server, and works with CRM systems. It requires mail-server configuration and does not provide a pre-send signature preview.

## Provisional technical conclusion

The publicly accessible documentation reviewed so far does **not** document a self-service API that SavvyOS can call with a user email address and receive rendered signature HTML. Consequently, SavvyOS should not assume an undocumented endpoint or store credentials until WiseStamp confirms an enterprise API/export option.

The supported integration most compatible with email generated by SavvyOS/Resend may be server-side deployment, but it needs explicit confirmation that WiseStamp can process messages emitted by Resend and preserve the sender's individual identity. If WiseStamp offers an enterprise API, the required contract should return user-scoped sanitized HTML (and ideally a version or updated timestamp) using a least-privilege service credential.

## Still to verify

- Whether WiseStamp exposes a documented enterprise API or supported signature-HTML export for third-party systems.
- Whether WiseStamp server-side routing can process Resend-originated emails and map the external From/Reply-To configuration to the correct user signature.
- Required WiseStamp plan, authorization model, setup/contact route, and API credential format.

3. [Deploy WiseStamp on CRM Platforms](https://support.wisestamp.com/en/articles/12561478-deploy-wisestamp-on-crm-platforms) explicitly supports server-side signatures for CRM-originated messages, but states that the organization must use **Google Workspace or Microsoft Exchange**. Its documented flow is CRM → organization email server → WiseStamp server-side host → organization email server → recipient.

## Refined conclusion

WiseStamp’s supported CRM route is **mail-flow insertion**, not an HTTP signature lookup. It is documented for organizations whose CRM messages travel through Google Workspace or Microsoft Exchange. SavvyOS currently sends directly through Resend, so this route will not work as a direct plug-in unless the organization changes the message path or WiseStamp provides a separate approved API/export capability.

## Required clarification from WiseStamp

Ask WiseStamp Sales/Support whether the Savvy STR Agents account includes, or can enable, a supported per-user HTML-signature retrieval/export API for a third-party CRM that sends through Resend. Also ask whether their server-side solution can accept mail from Resend directly; their public CRM guide describes Google Workspace or Microsoft Exchange as the required organization mail server.

4. [Deploy Your Signature Using Copy/Paste](https://support.wisestamp.com/en/articles/12561475-deploy-your-signature-using-copy-paste) documents a manual fallback: an assigned user can access WiseStamp’s Employee Hub and select the Copy/Paste path to copy their signature HTML. This is suitable only as an interim user-managed fallback, not as a reliable automatic synchronization mechanism.

## Interim fallback if no approved API is available

SavvyOS can provide a secure per-user “signature HTML” field where each user pastes the HTML copied from their WiseStamp Employee Hub. The system should sanitize it before storage, append it only to that user’s outbound emails, and allow the user to refresh it when WiseStamp changes. This preserves the requested email behavior without relying on an undocumented WiseStamp API.
